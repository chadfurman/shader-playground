Open Questions

Your concerns, captured. None of these are blockers — they're discovery work.

OQ1: What is "terrain"?

PROTOTYPE

Fractal flames are attractors, not terrain meshes. What does "flying through" them mean? Are flames nodes you fly between? Are you inside the attractor? Both? Resolved by: building the simplest flight prototype and seeing what feels right.

OQ2: What do player lights look like?

PROTOTYPE

A glowing point? A small flame? A particle trail? How does it read against fractal backgrounds? Resolved by: trying 3-4 visual treatments and playtesting.

OQ3: Multiple flames coexisting

R&D

shader-playground renders one genome fullscreen. Spanda needs multiple simultaneously. Spatial partitioning? Multiple render targets? Blended fields? This is a core rendering architecture question. Needs dedicated R&D spike.

OQ4: "Close to a flame" — proximity UX

PROTOTYPE

How does the player sense proximity? Visual glow? Audio resonance? Haptic pulse? Camera shift? The fractal pulling toward you? Resolved by: prototyping proximity feedback cues.

OQ5: Rendering performance

ARCHITECTURE

Fractals are expensive. Game logic + physics + multiple flames + networking = more expensive. Mitigations: LOD for distant flames, temporal reprojection, adaptive quality, render thread (already built). Architecture decisions, but the render thread gives us a strong starting point.

OQ6: Taste engine — implicit signals

R&D

Explicit votes are clean data. Movement-as-preference is noisy. How to filter signal from random movement? How fast should taste converge? How to blend tastes in multiplayer? Needs experimentation with the existing IGMM model + new implicit signal pipeline.

OQ7: Server security & architecture

LATER — phase 2+

Dedicated vs P2P? Genome validation (malicious data)? Payment processing? Privacy? Build single-player first. Multiplayer architecture is a separate design cycle once core gameplay is proven.
