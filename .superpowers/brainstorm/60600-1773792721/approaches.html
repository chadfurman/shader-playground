Three Approaches: How Do We Get There?

Same destination, different roads. Click the one that feels right.

A

Fork & Evolve

Copy shader-playground into a new repo. Build the game on top.

How it works: Full copy of shader-playground becomes the game's foundation. Strip the HUD/dev tooling, add game systems (flight, input, modes, networking). Shader-playground continues evolving separately as the art tool.

Architecture: Monolithic Rust binary, single repo. Game-specific code lives alongside the renderer/genome/audio systems. Platform abstraction via feature flags + cfg targets.

Pros: Fastest to first prototype. All existing code just works. Easy to experiment — change anything without worrying about breaking the art tool.

Cons: Diverges from shader-playground immediately. Bug fixes / improvements have to be manually ported back. Two codebases to maintain long-term.
B

Extract Core Library

Pull shared systems into a Rust library crate. Both projects depend on it.

How it works: Create flame-engine as a library crate containing: genome/genetics, audio analysis, taste model, flame compute pipeline, weights system. Shader-playground and the game are separate binaries that both depend on flame-engine.

Architecture: Cargo workspace with 3 crates: flame-engine (lib), shader-playground (bin), fractal-flight (bin). Shared WGSL shaders via build.rs. Platform layer abstracted behind traits in the engine.

Pros: Clean separation. Improvements to engine benefit both projects. Forces good API boundaries. Engine becomes reusable for future projects. Best for cross-platform — the engine is pure, apps handle platform details.

Cons: Upfront refactoring cost before you can prototype gameplay. Need to design the engine API before you fully know what the game needs. Risk of premature abstraction.
C

Fork First, Extract Later

Recommended. Start with A, graduate to B once the game design solidifies.

How it works: Copy shader-playground into fractal-flight/. Build rapid prototypes of flight, input modes, and game feel. Once you know what the game IS (maybe 2-4 weeks of prototyping), extract the shared engine into a library crate. Then both projects converge on it.

Architecture: Phase 1: monolithic fork, messy and fast. Phase 2: workspace extraction once APIs are clear. Phase 3: platform targets (web/mobile) added to the engine crate.

Pros: Fastest path to playable prototype. No premature abstraction. The engine API emerges from real usage, not speculation. You discover the game through play, THEN architect for it.

Cons: Some throwaway work during extraction. Requires discipline to actually do the extraction (tech debt gravity is real). Slight delay on cross-platform vs. B's cleaner start.

My Recommendation: C

You said "prototype and try stuff" three times now, and that instinct is correct. The worst thing we can do is spend weeks designing an engine API for a game we haven't played yet. Fork it, make it fly, discover the game through your hands, THEN clean up.

Prototype targets: 3 input modes (drift, tunnel, gestural) × fractal flight in ~1 week each. Whichever one makes you go "oh HELL yes" when you play it — that's the game.
